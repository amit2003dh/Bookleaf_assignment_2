## 1. Bubble.io Database Schema Design and Rationale

I decided to structure the Bubble.io database as a normalized, relational system rather than importing the sample JSON as a flat list. This prevents data duplication, ensures data integrity, and makes the system scalable as the number of authors and publications grows.

### Data Types and Field Configurations

#### A. User (Extends Bubble's Built-in User Type)
I modified the default User type to serve as the profile for both authors and administrators.
* Name (text): Full name of the author or admin.
* Phone (text): Contact number.
* City (text): City of residence (used for administrative filters).
* Joined_Date (date): The date the user joined the platform.
* Role (text): Determines portal authorization. Valid roles are "Author" or "Admin".
* Is_Processing_Notification (yes/no): A toggle field default set to "no" that tracks if an active royalty summary is running, used to implement UI idempotency.

#### B. Book
I created a separate Book table. Storing books as a separate data type allows authors to own multiple books and allows admins to query books independently of the user list.
* Book_ID (text): Unique book identifier (e.g., BK001).
* Title (text): Title of the publication.
* ISBN (text): International Standard Book Number.
* Genre (text): Genre of the book.
* Publication_Date (date): Date of publication (null for books in production).
* Status (text): Current publication status (e.g., "Published & Live", "In Production - Cover Design").
* MRP (number): Maximum Retail Price of the book.
* Author_Royalty_Per_Copy (number): Fixed royalty earned per sale.
* Total_Copies_Sold (number): Total copies sold.
* Total_Royalty_Earned (number): Total calculated royalty earnings.
* Royalty_Paid (number): Royalty amount already paid to the author.
* Royalty_Pending (number): Outstanding royalty balance.
* Last_Royalty_Payout_Date (date): Date of the last payout.
* Print_Partner (text): Print provider name (e.g., "In-House", "Repro India", "Epitome Books").
* Available_On (list of texts): Distribution platforms (e.g., Amazon, Flipkart, BookLeaf Store).
* Author (User): A relational reference linking the book back to the User record (Many-to-One relationship).

#### C. Notification_Log
I created a standalone table to track communication history without cluttering the User or Book tables.
* Author (User): Relational link to the author who received the summary.
* Triggered_By (text): Tracks the trigger source (e.g., "Admin Portal", "Bulk Process").
* Status (text): Current delivery status ("Processing", "Success", "Failed").
* Email_Content (text): The actual text content generated by the Google Gemini API.
* Timestamp (date): Time when n8n completed the callback.

## 2. Bubble Privacy Rules

To protect author data and comply with data governance standards, I configured backend privacy rules under the Data -> Privacy section in Bubble. 

I set the following access permissions:
* User: Administrators can view all fields and find users in searches. Authors can only view their own user fields. Everyone else has all access checkboxes unchecked, making user details invisible to unauthorized API queries.
* Book: Administrators can view all books. Authors can only search for and view books where "This Book's Author is Current User". Everyone else is blocked.
* Notification_Log: Administrators can view all logs. Authors can only view logs where "This Notification's Author is Current User". Everyone else is blocked.

This ensures data isolation, preventing an author from accessing another author's records through browser developer tools or network inspections.

## 3. Bubble to n8n Integration Protocol

The communication between Bubble and n8n is secure and bidirectional.

### A. Outgoing Webhook (Bubble to n8n)
When an admin clicks "Generate & Send", a Bubble page workflow triggers an API call configured in the API Connector:
* Method: POST
* Endpoint: The n8n webhook URL.
* Headers:
  - Content-Type: application/json
  - x-bookleaf-secret-token: secure_secret_token_123 (A shared secret key validated by n8n to ensure only authorized requests execute the workflow).
* Payload: The JSON body passes the author's unique ID, metadata, and their complete list of books formatted as a JSON array using Bubble's dynamic formatting.

### B. Inbound Callbacks (n8n to Bubble)
After processing the data and running the AI generation, n8n sends a REST update back to Bubble's Backend Workflows. These endpoints require a Bearer token generated in my Bubble settings:
* Success Path: n8n calls the "n8n_success_callback" endpoint, passing the "log_id" and the generated text. Bubble updates the Notification_Log status to "Success", saves the email content, and sets "is_processing_notification" back to "no".
* Failure Path: If any step fails, n8n redirects to "n8n_failure_callback", updating the log status to "Failed" with the error message and releasing the button lock.

## 4. AI Prompt Strategy and Policy Rules

I set up n8n to format the raw data inside a JavaScript Code node before sending it to the Google Gemini API (gemini-1.5-flash model). This ensures the LLM receives clean, structured data and reduces math errors.

### Prompt System Rules
I designed the prompt to enforce BookLeaf's specific corporate policies:
* Quarterly Payout Cycle: The prompt contains the schedule (Q1 paid by May 15, Q2 by August 15, Q3 by November 15, and Q4 by February 15 of the following year).
* Expected Payout Date: If the author's pending royalty balance is greater than zero, the AI calculates the next payout date based on the current quarter close and explains it.
* Acknowledging Completed Payments: If the pending amount is zero, the AI includes a positive acknowledgement.
* Books in Production: For books still in production (e.g. cover design or typesetting), the email explains that tracking begins after the book goes live.
* Tone: I configured the system prompt to maintain a warm, professional, and corporate tone.

## 5. Error Handling and Idempotency Controls

### Idempotency (Double-Click Prevention)
If an admin clicks "Generate & Send" twice, the system prevents duplicate triggers:
1. On click, the Bubble workflow immediately updates the Author's "is_processing_notification" field to "yes".
2. I configured a conditional rule on the button: "When Parent group's Author's is_processing_notification is yes -> This button is disabled and its text changes to 'Processing...'".
3. The button is re-enabled only when n8n executes the success or failure callback, setting the field back to "no".

### n8n Error Recovery
Inside the n8n workflow, I set the "OnError" property of the Gemini and SMTP nodes to "continueErrorOutput" and connected them to a Callback Failure node. If the Google API key fails or the mail server is unreachable, n8n will not stop silently; it will successfully update the log in Bubble to "Failed" with the error details.

## 6. My Learning Process and Challenges

As Bubble was new to me, I had to overcome a learning curve regarding its visual database architecture and container layout engine. Here are the key challenges I faced and how I solved them:

### A. Multi-Field Filter Application and Case Sensitivity
I encountered several challenges when implementing the search-and-filter logic in the Admin Portal:
* Case Sensitivity: I realized that Bubble's database-level searches are case-insensitive by default, but browser-side filtering using the ":filtered" operator is case-sensitive. To resolve this for partial name searches, I formatted both the Name field and the search input value to lowercase using the ":lowercase" modifier.
* Calculated Payout Status: Filtering the author list by "Royalty Status" (Paid, Pending, Overdue) was difficult because the status is not a simple database column but is calculated dynamically from the payout dates and pending balances of their books. I solved this by setting up conditional data sources on the Repeating Group, applying advanced filters that check the nested book list for aging thresholds.
* Empty Constraints: When a filter dropdown was left blank, Bubble originally returned zero results. I resolved this by checking "Ignore empty constraints" in the search query configurations.

### B. Data Alignment and Relational Joins
Coming from a SQL background, I struggled with the lack of direct database joins in Bubble. To align authors with their respective book lists:
* I mapped a "Many-to-One" relationship by adding an "Author" User field directly inside the Book type.
* I then used nested search queries inside my Repeating Groups, referencing "This Book's Author" to ensure that the sales, metadata, and book details aligned perfectly with the logged-in user without manual querying.

### C. Group Rendering and State Inspector Panels
Building the Admin Portal's split-screen view required displaying an author's full details only after clicking their row. 
* To implement this, I set up a custom state named "selected_author" on the page level.
* When a row is clicked, a workflow changes this state to the selected user.
* I set the detail container's data source to reference this custom state. This dynamic group rendering successfully populates the sidebar and shows the author's books without requiring separate page reloads.

### D. Flexbox Alignments and Responsive Spacing
I spent significant time learning Bubble's modern Flexbox layout engine:
* Centering Cards: For the login and signup card, I had to use an "Align to Parent" page layout, nesting a column group in the center with custom margins to keep it perfectly aligned on all screen sizes.
* Spacing and Margins: Getting the metrics cards row to align correctly required utilizing a "Row" container type with the space-between setting and configuring min-widths to prevent vertical squishing on smaller devices.
* Grid Layouts: Aligning headers with the repeating group rows was achieved by matching the margin, padding, and width percentage settings of the header columns directly with the elements inside the cell container.

### E. Dynamic JSON Construction and API Portability
* Inside the Bubble API Connector, I solved the challenge of sending lists of books as a JSON array by using the ":format as text" modifier, structuring the book parameters as raw JSON and using a comma as the delimiter.
* I resolved n8n node version mismatch warnings by replacing the native Google Gemini node with a standard HTTP Request node targeting the REST API endpoint, ensuring the workflow is portable across any n8n instance.
